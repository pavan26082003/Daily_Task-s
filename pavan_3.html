Things to Know

URL

The structure of a URL (Uniform Resource Locator) consists of several components, each serving a specific purpose. Here's the general structure:
protocol://username:password@hostname:port/path?query#fragment.

Explanation of Components:

1. **Protocol (Scheme)**:

Specifies the protocol used to access the resource. Examples include: - `http` (HyperText Transfer Protocol) - `https` (Secure HTTP) - `ftp` (File Transfer Protocol) - `mailto` (Email links)
Example: `https://`

2. **Username:Password (Optional)**:

Provides user credentials for accessing restricted resources.
Example: `username:password@`

3. **Hostname**:

Indicates the domain name or IP address of the server hosting the resource.
Example: `www.example.com`

4. **Port (Optional)**:

Specifies the port number to connect to on the server. If omitted, the default port is used based on the protocol (e.g., 80 for HTTP, 443 for HTTPS).
Example:`:8080`

5. **Path**:

Indicates the specific resource or file on the server. It can also include directories.
Example:`/folder/subfolder/file.html`

6. **Query (Optional)**:

Contains parameters passed to the server, typically in key-value pairs.
Example: `?key1=value1&key2=value2`

7. **Fragment (Optional)**:

Points to a specific part of the resource, often used in HTML documents to navigate to a section identified by an `id` attribute.
Example: Example URL https://www.example.com:443/folder/file.html?search=query#heading

Breakdown:

- Protocol: `https`
- Hostname: `www.example.com`
- Port: `443`
- Path: `/folder/file.html`
- Query: `?search=query`
- Fragment: `#heading`
This structure ensures URLs are standardized and allow web browsers and servers to interact effectively.


Software Development Methodologies

1. Agile Methodology

How it Operates: - Iterative and flexible approach with frequent delivery of working software. - Work is broken into small cycles (sprints) with continuous feedback.
**Best For:** - Dynamic projects where requirements may change (e.g., mobile apps, games). - Teams needing close collaboration and fast deliveries.

2. Scrum (Agile Framework)

**How it Operates:** - Uses short, time-boxed iterations called sprints (2–4 weeks). - Involves daily stand-ups, sprint planning, and retrospective meetings.
**Best For:** - Projects requiring frequent updates and rapid progress (e.g., e-commerce platforms). - Teams with defined roles like Scrum Master, Product Owner, and Development Team.

3. Waterfall Methodology

**How it Operates:** - Sequential and linear process where each phase (requirement → design → development → testing → deployment) must be completed before the next begins.
**Best For:** - Projects with fixed, well-defined requirements (e.g., government projects, basic static websites). - Simple and small-scale developments.

4. DevOps

**How it Operates:** - Integrates development and operations teams to automate testing, integration, and deployment. - Emphasizes Continuous Integration (CI) and Continuous Delivery (CD).
**Best For:** - Large-scale projects requiring frequent releases (e.g., SaaS platforms, cloud services). - Teams aiming for faster delivery and improved collaboration.

Summary:

- Agile: For dynamic and evolving projects.
- Scrum: For structured team collaboration and frequent delivery.
- Waterfall: For projects with stable requirements.
- DevOps: For frequent updates and automation.

Software Development Life Cycle (SDLC)

The Software Development Life Cycle (SDLC) is a systematic process used to design, develop, and maintain software applications. It provides a framework for ensuring software quality, meeting user requirements, and managing time and cost effectively.

Stages of SDLC

1. **Requirement Analysis**

- **Objective:** Gather and analyze the requirements from stakeholders.
- **Activities:** - Identify user needs and business goals. - Document functional and non-functional requirements.
- **Output:** Software Requirement Specification (SRS). ---

2. **Planning**

- **Objective:** Define the project scope, resources, timeline, and costs.
- **Activities:** - Create a project plan. - Assess risks and prepare mitigation strategies.
- **Output:** Detailed project roadmap. ---

3. **System Design**

- **Objective:** Define the system architecture and design based on the requirements.
- **Activities:** - Create high-level design (HLD) for overall structure. - Develop low-level design (LLD) for detailed components.
- **Output:** Design Document. ---

4. **Implementation (Coding)**

- **Objective:** Convert the design into actual software by writing code.
- **Activities:** - Use programming languages and tools to build the software. - Perform unit testing for individual modules.
- **Output:** Working software modules. ---

5. **Testing**

- **Objective:** Ensure the software meets requirements and is free of defects.
- **Activities:** - Perform functional, integration, system, and user acceptance testing (UAT). - Identify and fix bugs.
- **Output:** Tested and validated software. ---

6. **Deployment**

- **Objective:** Deliver the software to the production environment.
- **Activities:** - Deploy the software on servers or devices. - Provide user training and documentation.
- **Output:** Live software accessible to users. ---

7. **Maintenance**

- **Objective:** Ensure the software continues to perform optimally after deployment.
- **Activities:** - Fix bugs or issues reported by users. - Release updates or enhancements as needed.
- **Output:** Updated and well-maintained software.

Common SDLC Models

1. Waterfall Model: Linear, phase-by-phase approach.
2. Agile Model:Iterative, flexible with continuous delivery.
3. Iterative Model: Develop software incrementally in cycles.
4. V-Model:Testing is planned in parallel with development.
5. Spiral Model: Combines iterative development with risk analysis.

Benefits of SDLC

- Clarity: Well-defined stages ensure transparency and better communication.
- Quality Assurance: Regular testing ensures reliability and defect-free software.
- Cost-Efficiency: Early risk identification saves time and money.
- Adaptability: Models like Agile allow handling changing requirements.

When to Use SDLC

SDLC is used for any structured software project, from small applications to complex enterprise systems, ensuring consistent and predictable results.
